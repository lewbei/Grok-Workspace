# Release Checklist: UX Parity

Use this checklist before calling the web shell "Codex-window ready."

Status key:
- `PASS` validated and accepted
- `FAIL` known missing or broken
- `UNKNOWN` not yet checked

## Window Shell

- `FAIL` Left rail feels structurally similar to the Codex app window
- `PASS` Settings is pinned at the bottom-left of the sidebar
- `FAIL` Top chrome feels thinner and more desktop-like than the current function-first shell
- `FAIL` Conversation canvas has Codex-like proportions at desktop width
- `FAIL` Composer reads as a command surface instead of a basic form

## Navigation and Information Hierarchy

- `FAIL` A local project grouping layer exists above threads
- `FAIL` Threads can be scanned quickly through denser cards and stronger active state
- `FAIL` Project and thread affordances do not rely on fake or non-functional controls
- `UNKNOWN` Search/filter for threads is available and usable
- `UNKNOWN` Pinning or rapid access for recent work is available

## Transcript and Message Presentation

- `PASS` User prompt appears immediately
- `PASS` Assistant placeholder appears immediately
- `PASS` Streaming text is visible while the response is running
- `PASS` Tool traces are visible in the transcript
- `PASS` Usage and cost cards appear after completion
- `FAIL` Message spacing, typography, and badge density feel close to the Codex reference
- `FAIL` Markdown and code blocks have final-quality visual rhythm and hierarchy

## Settings and Disclosure

- `PASS` Settings is discoverable without searching the interface
- `PASS` Current API-key state is visible
- `PASS` Current mode and tool selections are visible in the composer and transcript
- `UNKNOWN` Empty-state guidance is polished enough for first-run users
